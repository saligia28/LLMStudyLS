# Step 49: MCP 实践｜整理“本地操作 Agent”总结

## 学习目标

这一节不是“再讲一个新知识点”，而是把前面六节收束成一份可执行的方法论：**如何把本地操作 Agent 做得稳定、可控、可审计、可扩展**。

学完这一节，你应该能：

1. 说出本地操作 Agent 的最佳实践
2. 明确它的能力边界和限制
3. 总结出一套适合持续迭代的工具治理规则
4. 理清下一周继续扩展 MCP 时该补什么

> **本节主线：** 从“会做工具”升级到“会治理工具系统”。

## 一、先给本周做一个整体回看

### 1.1 Week 7 的学习弧线

这一周其实是在搭一条完整链路：

1. 先理解 MCP 是什么
2. 再实现只读文件工具
3. 再实现文件写入工具
4. 再实现 shell 执行工具
5. 再把它们注册进统一 registry
6. 再把自动改码流程做成受控链路
7. 最后沉淀成本地操作 Agent 的实践总结

这条弧线的意义很明确：**从协议认知到工具实现，再到安全编排和工作流治理**。

### 1.2 这一周真正学到的不是“写了几个工具”

真正重要的是这些判断力：

- 哪些能力应该暴露为 tool
- 哪些上下文应该暴露为 resource
- 哪些动作必须加白名单
- 哪些操作必须二次确认
- 哪些失败必须回滚

这就是本周最核心的成长。

## 二、本地操作 Agent 的最佳实践

### 2.1 最小权限原则

本地操作 Agent 一定要先满足“够用”，再考虑“强大”。

建议默认策略是：

- 先给只读工具
- 再给写入工具
- 最后才给 shell

并且每一类工具都要有自己的权限范围，而不是统一开全权限。

### 2.2 工具要少而精

工具越多，不一定越强，往往越容易失控。

更好的方式是：

- 一个读取工具覆盖常见上下文获取
- 一个写入工具覆盖有限状态变更
- 一个 shell 工具只负责白名单命令

这样 Agent 更容易选择，也更容易审计。

### 2.3 所有副作用都要可见

凡是会改文件、跑命令、影响环境的动作，都应该记录：

- 谁发起的
- 调了哪个工具
- 用了什么参数
- 执行结果是什么
- 是否需要回滚

如果一条动作链路无法被复盘，那它就不适合自动化执行。

### 2.4 输出要结构化

不要只返回一句“执行成功”。

更好的结果应该包含：

- `ok`
- `stage`
- `changedFiles`
- `checks`
- `warnings`
- `audit`

结构化结果的价值在于：后续 Agent 可以继续基于它做下一步决策。

## 三、能力边界：本地操作 Agent 不能做什么

### 3.1 不能默认访问全盘

工作区外的文件、系统级目录、密钥、隐私配置，都应该默认拒绝。

### 3.2 不能默认执行任意 shell

shell 工具只能服务于白名单命令。只要有“任意命令执行”这个入口，Agent 就不再是局部自动化，而是系统级风险源。

### 3.3 不能跳过确认就直接落盘

尤其是：

- 多文件批量修改
- 覆盖已有逻辑
- 触碰配置、环境变量、脚本入口

这些动作都应该经过预览和确认。

### 3.4 不能把大上下文原样喂给模型

读文件不是“越多越好”，而是“越相关越好”。

建议始终控制：

- 单次读取大小
- 单次修改文件数
- 单次 shell 输出量

否则 Agent 很容易在噪声里迷失。

## 四、建议的 Agent 运行模板

### 4.1 一个稳定的执行节奏

```text
读上下文
→ 归纳目标
→ 生成计划
→ 提交修改方案
→ 人或策略确认
→ 执行修改
→ 运行验证
→ 记录审计
→ 输出总结
```

这个节奏的优点是：

- 不容易乱改
- 便于加回滚
- 便于加人工审核
- 便于后续扩展成多轮任务

### 4.2 建议的结果模板

```json
{
  "ok": true,
  "summary": "Completed local operation task",
  "changedFiles": ["src/app.js"],
  "checks": {
    "lint": "passed",
    "test": "passed"
  },
  "warnings": [],
  "nextSteps": [
    "Review the diff",
    "Merge if the checks stay green"
  ]
}
```

这种总结方式比“任务已完成”更适合作为 Agent 的长期工作接口。

## 五、下一周怎么衔接

### 5.1 这一周结束后，你已经具备什么

如果 Week 7 学完，你手里已经有了一个最小但完整的本地操作 Agent 能力栈：

- 能发现工具
- 能读上下文
- 能写受控修改
- 能执行白名单命令
- 能审计和回滚

这说明你已经从“会调用模型”进入“会治理执行系统”的阶段。

### 5.2 下一周最适合补什么

后续如果继续往前做，通常可以补这几类能力：

- 更丰富的 resource 管理
- 多工具协同和任务编排
- 更细粒度的权限控制
- 长任务状态管理
- 更完整的错误恢复和重试机制

换句话说，下一步不是再做一个孤立工具，而是把本周积累的能力组织成更完整的 Agent 工作台。

---

## 六、小结

Week 7 的核心结论可以压成一句话：

> **MCP 不是让模型“拥有更多权力”，而是让工具能力以更标准、更安全、更可控的方式接入模型。**

如果你把这一点记住了，那么后面不管是远程工具、跨服务编排，还是更复杂的自动化 Agent，你都会更容易守住边界，也更容易把系统做稳。
