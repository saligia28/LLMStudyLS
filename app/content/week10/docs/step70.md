## 二、Week 10 的核心结论

### 结论 1：长度不是越大越好，也不是越小越好

- 太大：主题混杂，向量变钝
- 太小：语义破碎，答案不完整
- 合适：一个 chunk 尽量覆盖一个局部主题

### 结论 2：overlap 是边界保险，不是万能药

- 有 overlap，边界信息更稳
- overlap 太大，重复和成本会上升
- 最好的 overlap，通常是“刚好补齐边界语义”

### 结论 3：结构切分通常优于纯字符切分

对于 Markdown、教程、规范类文档，标题、段落、列表、代码块本来就带着语义边界。先利用结构，再用长度约束，通常比单纯固定长度更自然。

### 结论 4：metadata 不是附加项，而是可用性的核心

没有 metadata，chunk 只能“被用”；有 metadata，chunk 才能“被解释、被过滤、被回放”。

### 结论 5：chunk 的好坏必须通过实验说话

主观感觉只能当起点，不能当结论。真正要保留的是：

- 问题集
- 对照组
- 指标
- 可复现的实验脚本

## 三、一个生产可用的默认策略

如果你现在要给一个新文档系统定默认值，可以先从这个思路开始：

```text
1. 先按标题、段落、代码块做语义切分
2. 再按 token 范围合并到目标长度附近
3. 在边界处加入适量 overlap
4. 每个 chunk 写入 docName / section / page / chunkIndex
5. 先跑一套小问题集，再决定是否调参数
```

### 一套很朴素但实用的默认值思路

| 文档类型 | 建议方向 |
| --- | --- |
| FAQ / 短说明 | 小 chunk，低 overlap |
| 长教程 / 文档 | 中等 chunk，适度 overlap |
| PDF / 章节书 | 结构切分优先，补页码 metadata |
| 代码文档 / API 手册 | 保留代码块与表格完整性 |

这不是固定标准，而是起步方案。真正的最优值，要靠你自己的实验去找。

## 四、Week 10 之后，你应该形成的工作习惯

### 1. 先看文档结构

不要一上来就按字数猛切。先问：

- 这是什么文档类型？
- 标题层级清不清楚？
- 有没有表格、代码、列表？
- 是否适合按章节切？

### 2. 先做小实验

不要一开始就追求“完美策略”。先拿 3 到 5 个 chunk 配置跑小实验，看 recall、MRR、覆盖率和成本。

### 3. 先保留可追溯性

任何 chunk 出问题，都要能回到原文。没有 metadata，后续调试会非常痛苦。

### 4. 先让工具可复用

Chunk 工具最好能支持：

- 不同策略
- 不同文档类型
- 不同参数
- 不同输出 metadata

这样后面接 RAG、知识库、Agent 时，不用每次重写。

## 五、和 Week 9 的关系

Week 9 让你理解了 embedding 和向量存储的基本能力；Week 10 让你理解了这些能力真正吃到的“原料”是什么。

可以把它们理解成：

```text
Week 9: 让文本进入可检索空间
Week 10: 让进入的文本足够像“可检索的好样本”
```

如果前者决定“怎么存”，那后者决定“存什么、存多大、带什么标签”。

---

## 六、最后的总结

Chunking 的方法论可以浓缩成 5 句话：

1. 先看语义边界，再定长度。
2. overlap 是补边界，不是堆内容。
3. metadata 决定可追溯性和可调试性。
4. 召回实验比直觉更可靠。
5. 工具要能稳定、复用、优化，而不是只在 demo 里好看。

如果把 Week 9 和 Week 10 合在一起看，你会得到一条完整的 RAG 入库逻辑：

```text
文档理解
  → chunking
  → metadata
  → embedding
  → vector store
  → retrieval
  → rerank
  → answer
```

这就是 Week 10 想让你真正带走的东西：不是某个参数值，而是一套能持续迭代的 Chunking 方法论。
