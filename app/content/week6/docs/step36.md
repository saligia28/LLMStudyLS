# Step 36: ReAct 多步 Agent｜理解 ReAct（计划状态 + 行动循环）模式

## 学习目标

这一节回答的核心问题是：**ReAct 为什么不是“多输出一点思考过程”，而是“让模型在计划状态和工具动作之间持续推进任务”？**

完成后你应该能：

1. 理解 ReAct 和普通工具调用的区别
2. 看懂“计划状态 → 工具动作 → 观察结果”的循环
3. 知道为什么不该把长篇外显思维链当作 ReAct 的重点
4. 明确 Agent 主线继续使用 `deepseek-chat`

> **本节默认模型**：工具型 Agent 继续使用 `deepseek-chat`。`deepseek-reasoner` 不放进工具循环。

---

## 一、ReAct 真正在解决什么问题

很多任务不是一次回答就结束，而是要：

- 先查，再比
- 先搜，再总结
- 先拿结果，再决定下一步

所以 ReAct 的重点不是“让模型多说几句想法”，而是让它每一轮都能基于新结果调整行动。

```text
用户问题
  ↓
计划状态：当前知道什么，还缺什么
  ↓
工具动作：执行哪个工具
  ↓
观察结果：工具返回了什么
  ↓
更新计划状态
  ↓
继续 or 结束
```

---

## 二、和普通 Function Calling 的区别

| 维度 | 普通 Function Calling | ReAct 风格 Agent |
| --- | --- | --- |
| 目标 | 触发一次工具调用 | 让任务在多轮中推进 |
| 重点 | 参数是否正确 | 下一步该做什么 |
| 状态 | 通常较短 | 需要持续维护计划状态 |
| 可观测性 | 看调用结果 | 看循环如何收敛 |

所以 ReAct 不应该被理解成“把完整思维链打印出来”，而应该被理解成“保留最小可观测状态，让系统知道下一步怎么走”。

---

## 三、推荐的最小 trace 结构

不建议把内部思考写成长篇文本，建议只保留足够调试的状态：

```js
const trace = [
  {
    step: 1,
    plan: '需要先查北京天气',
    action: { tool: 'searchWeather', input: '北京 今天天气' },
    observation: '北京，多云，8°C，风大',
  },
]
```

这个结构已经足够回答：

- 为什么调这个工具
- 工具返回了什么
- 下一轮为什么继续
