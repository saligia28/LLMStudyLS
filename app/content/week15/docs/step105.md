## 三、技术选型决策树

```text
你的 RAG 应用需要什么？
│
├── 快速上线，无需 GPU
│     └── DeepSeek API + Ollama（开发用）
│
├── 数据不能出机器
│     ├── 开发/小并发 → Ollama + q4_K_M
│     └── 生产/高并发 → vLLM + int8
│
├── 成本最低
│     ├── 有 GPU → vLLM（自托管，无 API 费用）
│     └── 无 GPU → Ollama（CPU 可跑，但慢）
│
├── 质量最高
│     ├── 有 API 预算 → DeepSeek / OpenAI
│     └── 本地优先   → vLLM fp16
│
└── 边缘设备 / MacBook 部署
      └── Ollama + qwen2.5:7b-q4_K_M
```

## 四、五周学习收获清单

**可以做到的事**：

- [ ] 构建完整的 RAG 系统（文档入库 → 检索 → 生成 → 引用）
- [ ] 把 RAG 封装成前后端分离的 web 应用
- [ ] 用 Ollama 在本地运行 llama3 / qwen2.5
- [ ] 启动 vLLM API 服务器，接入已有前端
- [ ] 对比 fp16 / int8 / int4 在速度和质量上的差异
- [ ] 写一个多后端适配器，一行改配置切换推理引擎
- [ ] 输出一份性能对比报告

**建立的认知**：

- RAG 的质量瓶颈通常在**检索层**，而不是生成层
- Ollama vs vLLM 不是好坏之分，而是**场景适配**
- 量化是"空间换精度"，工程师要知道在哪里值得换
- 生产系统需要**评估框架**，不能靠主观感受

## 六、小结

Week 11-15 是一个完整的从"RAG 原理"到"可部署产品"再到"推理优化"的学习闭环。

三句话总结：

1. **RAG = 知识检索 + 语言生成**，把两者分开优化，组合才有威力。
2. **本地模型不是"便宜的 GPT-4"**，而是隐私保护、低延迟、可定制的工具。
3. **量化是工程师的基本功**，它让资源有限的你也能用上更大的模型。

下一个阶段（Week 16+）将进入模型微调、评估和完整项目实战。
